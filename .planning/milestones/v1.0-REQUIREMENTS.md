# Requirements Archive: StoryNvim v1.0

**Archived:** 2026-04-12
**Core Value:** The config must stay easy to reason about while moving from one giant startup file to a clean modular structure that preserves current functionality.
**Milestone Outcome:** Shipped with known non-blocking tech debt

## v1 Requirements

### Bootstrap

- [x] **BOOT-01**: `init.lua` contains only minimal startup orchestration for leader setup, runtime bootstrap, and module entrypoints
  Outcome: validated in Phase 1 verification
- [x] **BOOT-02**: `lazy.nvim` bootstrap remains functional after the refactor
  Outcome: validated in Phase 1 verification
- [x] **BOOT-03**: Module loading order preserves required early globals and runtimepath setup
  Outcome: validated in Phase 1 verification

### Plugin Structure

- [x] **PLUG-01**: Plugin declarations are moved out of the monolithic `init.lua` into a `lua/plugins`-style import structure
  Outcome: validated in Phase 2 verification
- [x] **PLUG-02**: Plugin specs are grouped by concern so related plugins can be found and edited without scanning the whole config
  Outcome: validated in Phase 2 verification
- [x] **PLUG-03**: Existing optional plugin modules remain compatible or are migrated consistently into the new structure
  Outcome: validated in Phase 4 verification

### Core Config

- [x] **CORE-01**: Editor options are defined in dedicated modules instead of inline in `init.lua`
  Outcome: validated in Phase 3 verification
- [x] **CORE-02**: Keymaps are defined in dedicated modules instead of inline in `init.lua`
  Outcome: validated in Phase 3 verification
- [x] **CORE-03**: Autocommands and other startup helpers are defined in dedicated modules instead of inline in `init.lua`
  Outcome: validated in Phase 3 verification

### Behavioral Parity

- [x] **PAR-01**: Core editing behavior remains unchanged from the user perspective after the migration
  Outcome: validated in Phase 4 verification
- [x] **PAR-02**: LSP, completion, formatting, Treesitter, and Telescope still load and work after the migration
  Outcome: validated in Phase 4 verification
- [x] **PAR-03**: Health checks and optional plugin modules still load correctly after the migration
  Outcome: validated in Phase 4 verification

### Documentation

- [x] **DOC-01**: Repository documentation describes the modular layout and where to customize the config
  Outcome: validated in Phase 5 verification
- [x] **DOC-02**: The repo no longer presents the single-file layout as the intended architecture for this fork
  Outcome: validated in Phase 5 verification

## Deferred To Next Milestone

- **FUT-01**: Add automated smoke checks or tests for critical Neovim startup behavior
- **FUT-02**: Revisit optional module boundaries once the main migration is stable

## Out of Scope At v1.0 Ship Time

| Feature | Reason |
|---------|--------|
| Swapping to a different plugin manager | Not required for the structural migration |
| Broad plugin churn unrelated to modularization | Increases regression risk and obscures refactor goals |
| Turning the repo into a full Neovim distribution | The project goal is maintainable personal structure, not distro design |
